# Prompt Craft v2.0 架构设计规范（SPEC）

> **版本**：v2.0（重构版）
> **目标**：从"简单的6段式模板"升级为"可组合、可验证、可扩展的提示词架构"
> **本文件为设计规范，执行请参考末尾的实现计划**

---

## 第一部分：问题诊断（Why v2.0?）

### v1.0 架构评估

v1.0 的核心结构是 6 段式：角色设定 → 项目架构 → 核心功能 → 设计约束 → 输出要求 → 最佳实践。

**优点**：
- ✅ 简单易懂，快速上手
- ✅ 有明确的角色设定（Role Prompting）
- ✅ 覆盖了技术栈和功能范围
- ✅ 有约束条件，避免生成偏离方向

**缺点**：

| 问题 | 影响程度 | 具体描述 |
|------|---------|---------|
| 🔴 **结构标准不统一** | 高 | 不同模板的字段颗粒度不一致，有的有"设计约束"有的隐含在"项目架构"中 |
| 🔴 **缺少负面约束** | 高 | 仅有"最佳实践"（做什么），缺少"反模式清单"（不要做什么） |
| 🔴 **缺少验证机制** | 高 | 没有告诉 AI "生成后应该如何自检"，导致生成的代码可能有明显缺陷 |
| 🔴 **没有验收标准** | 高 | "输出要求"太模糊，没有量化的 Acceptance Criteria |
| 🟡 **技术栈单一化** | 中 | 每个场景只推荐一套方案，没有 A/B 选项供用户选择 |
| 🟡 **最佳实践太抽象** | 中 | "使用 Redis 缓存"这种描述不够具体，应该是"热点数据 TTL = 15min，冷数据缓存穿透保护" |
| 🟡 **没有测试要求** | 中 | 后端模板完全不提单元测试/集成测试要求 |
| 🟡 **没有场景化提示** | 中 | 没有区分"快速原型"和"生产级项目"，两套方案混在一起 |
| 🟢 **可执行性不足** | 低 | "添加详细中文注释"这种要求，不具体到文件类型和注释格式 |

### 真实使用中的问题（基于大屏生成的反思）

在刚才的大屏生成中，AI 实际产出的代码与理想状态的差距：

1. **布局问题**：地图被设置为 `grid-row: span 2`，但网格只有 2 行，导致地图占据了整个右列，但左列的 4 个图表（趋势图、饼图、柱状图、订单列表）反而显得拥挤
2. **地图依赖**：使用了 echarts 的 china.js，该文件在某些环境中不可用，且新版 echarts 已改为需要单独注册地图 JSON
3. **模拟数据**：数据完全随机，没有体现真实的业务逻辑（如工作日 vs 周末、促销活动影响等）
4. **交互缺失**：模板中要求"数据下钻、图表联动"，但生成的代码没有任何点击/悬停交互

**根本原因**：提示词中的"设计约束"和"输出要求"不够具体，没有强制执行机制。

### 核心设计理念：PCTF+ 架构

├─────────────────────────────────────────────────────────────┤

│   F — Format（格式/验收）    ← 质量：输出格式+验收标准           │
│                                                               │
│              ┌─ 🔴 Negative Constraints（负面约束）            │
│   增强层 +   ├─ 🧩 Optional Modules（可选插件模块）             │
│              ├─ ✅ Validation Checklist（验证清单）             │
│              ├─ 🧪 Testing Requirements（测试要求）            │
│              └─ ⚠️ Common Pitfalls（常见陷阱）                 │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### 标准模板的 10 大字段（黄金结构）

```
1.  🎭 角色设定（Persona）
    ├── 专业身份（如"10年经验的前端可视化专家"）
    ├── 核心能力（技能栈 + 业务理解）
    └── 工作风格（如"优先考虑性能和可维护性，而非炫技"）

2.  🏗️ 项目架构（Context）
    ├── 技术栈（A方案 / B方案 可选）
    ├── 目录结构（明确层级，支持不同规模）
    └── 代码规范（命名、格式化、lint配置）

3.  🎯 核心功能需求（Task）
    ├── Must Have（必须实现的功能）
    ├── Should Have（建议实现的功能）
    └── Could Have（可省略，根据需求可选）

4.  📐 设计约束（Constraints）—— 必须量化！
    ├── 性能约束（如 "首屏 < 1.5s，LCP < 2.5s"）
    ├── 交互约束（如 "所有可点击元素应有3种状态：default/hover/active"）
    ├── 兼容性约束（如 "支持 iOS 13+ / Android 8+"）
    └── 视觉约束（如 "主色 #00d4ff，文字对比度 > 4.5:1"）

5.  📤 输出要求（Output Format）—— 必须精确！
    ├── 文件结构（列出文件路径和用途）
    ├── 代码风格（Vue SFC / React FC / 原生JS）
    ├── 注释规范（行内注释 + 函数JSDoc + 复杂逻辑解释）
    └── 特殊要求（如 "支持直接在浏览器中打开，无需构建"）

6.  ✅ 验收标准（Acceptance Criteria）—— 必须可验证！
    ├── Given-When-Then 格式
    └── 每条标准都应能被人工或自动化测试验证
    示例：
    - Given 用户在桌面浏览器打开页面
    - When 窗口大小从 1920px 调整到 1280px
    - Then 所有图表应自适应，无内容溢出和错位

7.  🚫 反模式清单（Anti-Patterns / What NOT to do）
    ├── 明确禁止的行为
    ├── 常见错误
    └── 性能/安全/可维护性的禁忌

8.  ⚠️ 常见陷阱与规避（Common Pitfalls）
    ├── 该场景开发者常犯的错误
    ├── 具体的规避方法
    └── 代码示例（坏代码 vs 好代码）

9.  🧪 测试要求（Testing Requirements）
    ├── 单元测试覆盖率目标
    ├── 关键路径的集成测试
    └── 手动测试清单

10. 💡 最佳实践与优化建议（Best Practices）—— 必须可执行！
    ├── 按照优先级排序
    ├── 每条建议都应该可直接转化为代码行动
    └── 可选：参考链接/参考项目
```

### 设计原则

**原则 1：约束必须量化**
- ❌ 不要写："应该有良好的性能"
- ✅ 应该写："首屏 LCP < 2.0s，JS 总大小 < 300KB（gzip）"
- ✅ 应该写："所有 API 请求 P95 < 200ms，超时时间 5s，失败重试 2 次"

**原则 2：负面约束与正面约束同等重要**
- 告诉 AI "不要做什么"与"要做什么"同样有效
- 每个场景至少列出 3-5 条负面约束

**原则 3：验证驱动开发**
- "怎么验证这个功能是对的？" 是高质量 prompt 的灵魂
- 每条验收标准都应该可执行（可以手动检查，也可以自动化）

**原则 4：分档策略**
- **快速原型档**：只实现 Must Have，简化架构，Mock 数据
- **生产级档**：完整实现 Must + Should，加测试、加监控、加错误处理
- 用户选择档位后，AI 自动调整提示词的严格程度

## 📐 设计约束（Constraints）

### 性能约束（可量化）
- 首屏 LCP < 2.0s（1920x1080, Chrome, 4G 网络）
- JS 总大小 < 300KB（gzip 后）
- 单个图表渲染时间 < 100ms
- 页面无明显卡顿（60fps）

### 交互约束
- 所有可点击元素必须有 3 种视觉状态：default / hover / active
- 加载状态：所有异步操作必须有 loading 指示
- 错误状态：API 失败必须有用户可见的错误提示和重试按钮
- 移动端：触摸目标 >= 44x44px，防止误触

### 兼容性约束
- 浏览器：Chrome 90+ / Firefox 90+ / Safari 14+ / Edge 90+
- 分辨率：1920x1080（主）/ 2560x1440（高DPI）/ 1280x720（最小兼容）
- 移动端可选：1080x1920 / 750x1334

### 视觉与设计系统约束
- 主色调：#00d4ff （科技蓝）
- 背景色：#0a0e27（深空蓝）
- 文字色：主文字 #ffffff，次要文字 #8b9cbd
- 警示色：成功 #52c41a / 警告 #faad14 / 危险 #ff4d4f
- 字体：正文 system-ui / 数字 Menlo, monospace
- 圆角：4px / 8px / 12px 三级
- 间距：基于 4px 栅格系统（4, 8, 12, 16, 20, 24, 32）

### 安全约束
- 所有用户输入必须 XSS 过滤
- API 密钥不在前端硬编码
- 敏感数据传输使用 HTTPS

## ✅ 验收标准（Acceptance Criteria）

格式：Given-When-Then

- Given 用户在 1920x1080 的浏览器中打开页面
  When 等待页面完全加载
  Then LCP < 2.0s，且所有内容在 3 秒内可见

- Given 页面已加载完成
  When 用户调整浏览器窗口至 1280x720
  Then 所有内容自适应布局，无横向滚动条和内容溢出

- Given 图表数据正在加载
  When 用户看到加载动画
  Then 动画应平滑且有明确的加载进度指示

- Given 鼠标悬停在图表数据点上
  When 系统展示 tooltip
  Then tooltip 应在 100ms 内出现，且不被页面边界截断

- Given 用户连续刷新页面 10 次
  When 每次都应正常展示数据
  Then 页面不应出现白屏或 JS 报错

## 🚫 反模式清单（What NOT to Do）

请严格避免以下行为：

- ❌ **不要使用 any 类型**：除非与第三方库集成必须，否则必须有具体类型
- ❌ **不要硬编码配置**：颜色、尺寸、时间等应作为常量定义在配置文件中
- ❌ **不要忽略错误处理**：所有 API 调用、Promise、用户输入都必须有错误处理
- ❌ **不要使用 setTimeout 作为状态同步机制**：使用 Promise、事件或状态管理
- ❌ **不要在循环中操作 DOM**：使用 DocumentFragment 或虚拟列表优化
- ❌ **不要重复发明轮子**：已有成熟库/组件的功能，优先使用而非自己实现
- ❌ **不要使用过时的 API**：如 document.write、innerHTML（除非必要）、eval

## ⚠️ 常见陷阱与规避（Common Pitfalls）

### 陷阱 1：大屏尺寸适配问题
**问题**：设计时只考虑了 1920x1080，但实际大屏可能是 2K/4K 或拼接屏

**规避方法**：
```javascript
// 坏代码：固定像素
width: 1920,

// 好代码：使用百分比 + rem
width: '100%',  // 让容器自适应父元素
// 或者使用 CSS transform: scale() 实现整体缩放
```

### 陷阱 2：ECharts 地图依赖问题
**问题**：echarts 5.x 版本不再内置中国地图，需要手动注册 JSON

**规避方法**：
- 明确使用 `echarts.registerMap('china', chinaGeoJson)` 注册地图
- 或使用 DataV.GeoAtlas 的公开 API 获取地图数据
- 或使用纯数值型图表（如柱状图 + 省份名）避免地图依赖

### 陷阱 3：WebSocket 断线重连
**问题**：大屏展示通常 7x24 小时运行，网络波动会导致数据停止更新

**规避方法**：
```javascript
// 坏代码：无重连
const ws = new WebSocket(url);

// 好代码：指数退避重连
let retryCount = 0;
const MAX_RETRY = 10;
function connect() {
  const ws = new WebSocket(url);
  ws.onclose = () => {
    if (retryCount < MAX_RETRY) {
      const delay = Math.min(1000 * Math.pow(2, retryCount), 30000);
      setTimeout(connect, delay);
      retryCount++;
    }
  };
}
```

### 陷阱 4：内存泄漏
**问题**：长时间运行的大屏容易因定时器/事件监听/图表实例未清理而内存泄漏

**规避方法**：
- 所有 setInterval 必须有对应的 clearInterval
- ECharts 实例必须在页面卸载时调用 `dispose()`
- 事件监听必须配对使用 addEventListener 和 removeEventListener

## 第五部分：可选插件模块（Plugin Modules）

### 插件设计理念

用户的需求往往包含跨场景的共性需求（如"我需要国际化"、"我需要 Docker 部署"）。这些需求不属于任何单一模板，但可以附加到任意模板上。

**实现方式**：用户确认具体场景后，可以选择是否附加以下插件模块。AI 会将插件内容自动合并到提示词中。

### 插件清单

| 插件名称 | 适用场景 | 提供的内容 |
|---------|---------|-----------|
| 🌐 i18n 国际化 | 所有前端/全栈项目 | 多语言方案、翻译文件结构、语言切换逻辑 |
| 🐳 Docker 部署 | 后端/全栈项目 | Dockerfile、docker-compose.yml、镜像优化建议 |
| 🔐 用户认证 | 所有需要登录的项目 | JWT / OAuth / Session 方案，登录注册流程 |
| 📊 监控与日志 | 生产级项目 | Prometheus 指标、ELK 日志、APM 接入 |
| 📱 PWA 增强 | 移动端/H5 项目 | Service Worker、离线缓存、添加到主屏幕 |
| ♿ 无障碍（A11y）| 公共服务/政府网站 | ARIA 标签、键盘导航、屏幕阅读器兼容 |
| 🔄 CI/CD 流水线 | 生产级项目 | GitHub Actions / GitLab CI 配置、自动部署 |
| 🔒 安全加固 | 生产级项目 | CSRF/XSS/CSRF 防护、CSP、依赖检查 |
| 🎨 UI 库集成 | 前端项目 | Ant Design / Element Plus / shadcn/ui 集成方案 |
| ⚡ SSR/SSG 优化 | 内容型网站 | Next.js / Nuxt SSR 配置、ISR 策略 |

### 插件示例（Docker 部署插件）

```markdown
== 🐳 Docker 部署插件 ==

请在项目中添加以下内容：

1. Dockerfile（多阶段构建）:
   - Builder 阶段：安装依赖 + 构建
   - Runner 阶段：仅复制构建产物，使用轻量基础镜像
   - 目标镜像大小 < 200MB

2. .dockerignore:
   - node_modules, .git, dist, build, npm-debug.log

3. docker-compose.yml:
   - 包含 app 服务 + 依赖服务（如 PostgreSQL + Redis）
   - 合理的健康检查配置（healthcheck）

4. 环境变量:
   - 所有配置通过环境变量传入（使用 --env-file 或 .env）
   - 提供 .env.example 模板（不要包含真实密钥）
```

## 第六部分：分档策略（Profile Selection）

### 两档模式说明

**模式 A：🚀 快速原型（Rapid Prototype）**
- 目标：最快速度看到效果，便于演示和讨论
- 实现范围：仅 Must Have
- 代码质量：可工作即可，允许一定的技术债务
- 测试：手动测试主要流程即可
- 时间预期：< 1 小时生成

**模式 B：🏭 生产级（Production Grade）**
- 目标：可投入生产使用，代码质量高，可维护性好
- 实现范围：Must Have + Should Have
- 代码质量：严格遵循规范，完整的错误处理和类型定义
- 测试：完整的手动测试清单 + 自动化测试框架
- 时间预期：< 4 小时生成

### 分档实现机制

用户选择场景后，询问：
> "你需要【🚀 快速原型】（快速看效果）还是【🏭 生产级】（高质量可部署）？"

然后在提示词注入时动态调整：
- 快速原型：跳过 P1/P2 最佳实践，简化架构，Mock 数据优先
- 生产级：完整应用所有 10 个字段，严格遵循约束和验收标准

## 第七部分：使用流程（User Flow v2.0）

### 改进后的 6 步流程

```
Step 1: 意图识别（与 v1 相同）
   ↓
Step 2: 展示分类菜单（与 v1 相同）
   ↓
Step 3: 用户选择具体场景
   ↓
Step 4: 档位选择 + 插件选择  ← 新增
   │  "🚀 快速原型 还是 🏭 生产级？"
   │  "需要添加什么插件吗？（如 Docker 部署、i18n 国际化）"
   │  "有什么技术栈偏好吗？否则使用推荐方案"
   │  "补充任何特殊需求？"
   ↓
Step 5: 注入完整 v2.0 提示词（10大字段+插件+分档调整）← 核心改进
   ↓
Step 6: AI 生成代码 → 用户迭代反馈
```

### Step 5 的具体实现：提示词动态组装

```
最终提示词 = 基础模板(10字段)
           + 档位调整(快速原型/生产级)
           + 插件模块(用户选择的所有插件)
           + 用户自定义补充
```

复杂度: 中高
平均代码量: 800-1500行
预估开发时间: 快速原型 30min / 生产级 3h

## 第十部分：验证与质量保证

### 模板质量检查清单

在提交/使用任何模板前，检查以下内容：

- [ ] 10 大字段完整齐全
- [ ] "设计约束"中的性能要求都是可量化的（有具体数字）
- [ ] "验收标准"使用 Given-When-Then 格式，每条都可验证
- [ ] "反模式清单"至少 5 条
- [ ] "常见陷阱"至少 3 条，且每条都有"坏代码 vs 好代码"示例
- [ ] "最佳实践"按优先级排序（P0/P1/P2）
- [ ] 文件头部有完整的元数据（版本、分类、技术栈等）
- [ ] "测试要求"包含手动测试清单
- [ ] "核心功能需求"按 Must/Should/Could 分级
- [ ] 技术栈提供 A/B 两个方案

### 持续优化机制

1. **使用反馈收集**：每次使用后记录"生成的代码实际质量如何"
2. **反模式积累**：每次发现 AI 生成的不理想代码，总结为新的反模式
3. **版本迭代**：每 1-2 个月根据积累的反馈更新模板内容

## 总结

Prompt Craft v2.0 的核心改进：

| 改进方向 | v1.0 | v2.0 |
|---------|------|-------|
| 模板字段 | 6段式（角色/架构/功能/约束/输出/最佳实践） | 10段式 + 插件系统 |
| 设计约束 | 定性描述为主，部分量化 | **强制量化**，可测量可验证 |
| 负面约束 | 仅在主文档提及，模板中无实际应用 | **反模式清单**，每个模板至少5条 |
| 验证机制 | 无 | **验收标准（Given-When-Then）+ 常见陷阱代码对比** |
| 测试要求 | 无 | **手动测试清单**（可选自动化测试） |
| 场景分档 | 单档 | **🚀 快速原型 / 🏭 生产级** 两档可选 |
| 技术栈选项 | 单方案 | **A方案（推荐）/ B方案（替代）** |
| 插件系统 | 无 | **10+ 可选插件**（Docker/i18n/认证/监控等） |
| 最佳实践 | 简单列表 | **按 P0/P1/P2 优先级排序**，可直接转化为代码 |
