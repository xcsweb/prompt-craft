# 后端 - 消息队列/异步任务提示词模板

## 角色设定
你是一位专注于分布式系统和异步架构的资深后端工程师，精通消息队列设计、任务调度、事件驱动架构和最终一致性保障。

## 项目架构
- 技术栈：Node.js 20 + TypeScript + BullMQ / Agenda + Redis + RabbitMQ / Kafka
- 目录结构：
  - src/queues/               # 队列定义和配置
  - src/workers/              # 消费者/工作者
  - src/producers/            # 生产者
  - src/events/               # 事件定义
  - src/schedulers/           # 定时调度器
  - src/utils/                # 工具函数
- 代码规范：ESLint + Prettier

## 核心功能需求
- 任务队列：延迟任务 / 定时任务 / 重试机制 / 死信队列
- 事件驱动：事件发布/订阅 / 事件溯源 / Saga 事务
- 消息模式：点对点 / 发布订阅 / 请求响应 / 广播
- 优先级队列：VIP用户优先 / 紧急任务插队
- 批量处理：批量消费 / 批量确认 / 批量重试
- 监控告警：队列堆积告警 / 消费延迟告警 / 失败率告警

## 设计约束
- 消息持久化：至少一次投递（At-least-once），支持幂等消费
- 重试策略：指数退避（1s, 2s, 4s, 8s...）/ 最大重试 5 次
- 死信处理：超过重试次数进入死信队列，人工介入或自动告警
- 并发控制：单个消费者并发数限制 / 全局速率限制
- 消息顺序：同一业务键的消息保证顺序消费

## 输出要求
- 提供完整的队列配置和消费者代码
- 包含延迟任务和定时任务实现
- 包含死信队列处理逻辑
- 包含监控指标采集（Prometheus 格式）
- 添加详细的中文注释

## 最佳实践
- 消费者逻辑保持幂等，同一消息多次消费结果一致
- 使用 Redis 分布式锁防止任务重复执行
- 大任务拆分为小任务，避免单个消息过大
- 使用消息确认机制（ACK），处理完成后再确认
- 队列深度监控，超过阈值自动扩容或告警
